# Capture:D 项目开发规范（Single Source of Truth）

> **本文件是 Capture:D 项目唯一权威开发规范。**
> 所有开发决策、代码编写、架构调整必须以本文件为准。
> 规则变更只能追加，不能覆盖历史原则。每次追加需标注日期和原因。

## 一、项目目标

### 1.1 产品定位

**App 名称：Capture:D**

一款 iOS 原生 app，定位为手机里的"第二个照片 app"。用户通过系统分享功能将截屏或保存的图片归档到独立收藏库，按需手动触发 AI 识别出处并生成分类说明。与系统照片 app 完全隔离，独立存储空间，不访问系统相册。

### 1.2 核心功能

1. **图片归档**：通过 Share Extension 或 App 内 FAB 按钮将图片存入 app，发布时必须选择大分类（找书/找诗/找画/找歌），可选填命名和标签
2. **分类管理**：四个硬编码大类（找书/找诗/找画/找歌）+ 碎碎念，每个大类下有 `未整理/` 子文件夹
3. **碎碎念**：纯文字直接归为碎碎念，带图片则要求用户选择大分类
4. **AI 分析**：用户手动触发，AI 识别图片出处并生成分类说明（完整歌词/找诗全文/找书简介/找画分析）
5. **内容浏览**：双列瀑布流首页 + 纯文字详情页（逐条编辑+追加） + 左滑浮层卡片看原图
6. **内嵌搜索**：搜索在首页内进行，支持搜索+分类组合过滤
7. **跨分类关联**：一张图可属于多个分类，金色光晕提示 + tag 跳转
8. **费用管控**：AI 完全手动触发，用户可设月度分析上限

### 1.3 技术栈

| 项目 | 选型 |
|------|------|
| 平台 | iOS 18+（仅 iPhone） |
| 语言 | Swift |
| UI 框架 | SwiftUI |
| 本地存储 | SwiftData |
| 图片存储 | App 内独立存储 |
| 共享数据 | App Group |
| AI 接入 | 多模型（Claude / DeepSeek / 豆包），用户自填 API Key |
| API Key 存储 | iOS Keychain |
| 图片入口 | Share Extension + App 内 FAB 按钮（带图发布） |
| 显示模式 | 仅浅色模式 |

## 二、开发原则

### 2.1 核心原则

1. **正确性优先**：代码必须正确运行，不追求炫技
2. **可维护性优先**：代码清晰明确，程序员可读可修复，一个文件夹只干一件事
3. **小步迭代**：每个模块做完、测完、文档更新完才进下一个
4. **单模块开发**：一次只开发一个模块，禁止同时开发多个模块
5. **费用安全**：AI 绝不自动运行，所有 API 调用必须由用户手动触发

### 2.2 禁止事项

- **禁止 Vibe Coding**：不允许凭感觉写代码，每行代码必须有明确目的
- **禁止一次生成整个项目**：必须按模块逐步开发
- **禁止跳步开发**：分析→设计→实现→测试→文档→提交，不得跳过任何阶段
- **禁止 AI 自动触发**：AI 在用户点击按钮之前完全静默，不开机，不做任何准备
- **禁止访问系统相册**：App 不请求相册权限，完全独立

## 三、开发流程

### 3.1 流程阶段（严格按序执行）

```
分析 → 设计 → 实现 → 测试 → 文档更新 → Git 提交 → 等待确认
```

**每个阶段的详细要求：**

#### 阶段 1：分析
- 阅读 PROJECT_RULES.md、ARCHITECTURE.md、DEVELOPMENT_LOG.md
- 理解当前模块在整体架构中的位置
- 列出该模块的输入、输出、依赖关系
- 识别潜在风险

#### 阶段 2：设计
- 画出模块内部结构（文件列表、类/结构体、函数签名）
- 定义数据模型（如涉及 SwiftData）
- 定义接口（如与其他模块交互）
- 输出设计文档，等待确认

#### 阶段 3：实现
- 按设计文档编写代码
- 遵守代码规范（见第五章）
- 一个文件只做一件事
- 代码中不出现硬编码的字符串（使用 Constants）

#### 阶段 4：测试
- 编写单元测试，覆盖核心逻辑
- 手动测试 UI 交互
- 测试异常场景（无网络、AI 失败、空数据等）
- 所有测试必须通过

#### 阶段 5：文档更新
- 更新 ARCHITECTURE.md（如有架构变化）
- 更新 DEVELOPMENT_LOG.md（记录本次开发内容）
- 更新模块 README（如有）
- 如涉及重大决策，新建 ADR 文件

#### 阶段 6：Git 提交
- 提交信息遵守 Git 规范（见第八章）
- 只提交与当前模块相关的文件
- 不提交临时文件、.DS_Store 等

#### 阶段 7：等待确认
- 开发者审阅代码和文档
- 确认通过后才能进入下一个模块
- 发现问题则回到对应阶段修复

### 3.2 开始开发前（每次新对话必读）

```
首先阅读：
1. 项目大纲（PROJECT_SPEC.md）
2. CaptureD/PROJECT_RULES.md
3. CaptureD/ARCHITECTURE.md
4. CaptureD/DEVELOPMENT_LOG.md
5. CaptureD/ADR/*

阅读完成后，不要写代码。

先进行架构评审（Architecture Review），输出：
- 项目理解
- 核心模块
- 模块依赖关系
- 潜在风险
- 建议优化项
- 推荐开发顺序
- 是否发现架构问题

然后等待确认。
```

## 四、模块开发规则

### 4.1 核心规则

- **一次只能开发一个模块**
- **模块完成后才能进入下一模块**
- **禁止同时开发多个模块**

### 4.2 模块完成标准

一个模块被视为"完成"必须满足以下全部条件：

1. **功能完成**：所有设计的功能可正常运行
2. **测试通过**：单元测试全部通过，手动测试无明显 bug
3. **README 更新**：模块内有 README.md 说明模块职责和使用方式
4. **DEVELOPMENT_LOG 更新**：记录开发内容、决策原因、遇到的问题
5. **ARCHITECTURE 更新**：如模块涉及架构变化，更新架构文档
6. **Commit Message 生成**：符合 Git 规范的提交信息

### 4.3 模块划分

基于项目架构，模块开发顺序建议（最终顺序在架构评审后确认）：

| 优先级 | 模块 | 路径 | 职责 |
|--------|------|------|------|
| P0 | 数据模型 | `Core/Database/` | SwiftData 模型定义 |
| P0 | 图片存储 | `Core/Storage/` | 图片文件读写、空间计算 |
| P0 | Share Extension | `ShareExtension/` | 分享面板 UI + 图片写入 App Group |
| P1 | 首页瀑布流 | `Features/Collection/` | 双列瀑布流 + 分类 tab + 未整理入口 |
| P1 | 文件详情页 | `Features/Detail/` | 纯文字内容 + 左滑图片浮层 |
| P1 | 未整理文件 | `Features/UnsortedFiles/` | 未整理列表 + AI 分析触发按钮 |
| P2 | AI 接入层 | `Core/AI/` | 多模型适配 + Prompt 管理 |
| P2 | API Key 管理 | `Core/Keychain/` | Keychain 读写 |
| P2 | 设置页 | `Features/Settings/` | AI 配置 + 费用管理 + 存储 |
| P3 | 内嵌搜索 | `Features/Collection/` | 搜索内嵌至首页，搜索+分类组合过滤 |
| P3 | 回收站 | `Features/RecycleBin/` | 删除 + 恢复 + 30 天自动清理 |
| P3 | 首次引导 | `Features/Onboarding/` | 2-3 页引导流程 |
| P4 | 全局样式 | `Shared/Theme/` | AppTheme + 按钮/卡片/文字样式 |
| P4 | 通用组件 | `Shared/Components/` | 时间隔断、缩略图卡片等 |
| P4 | 动画系统 | 集成在 Collection | Safari 弹性动效 + 碎片化消失 |

## 五、代码规范

### 5.1 文件组织

- 每个 `.swift` 文件只做一件事
- 文件名与主要类型名一致（如 `AIManager.swift` 包含 `AIManager` 类）
- 文件按功能分组到对应的 Feature 或 Core 文件夹

### 5.2 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 类/结构体 | UpperCamelCase | `AIServiceProtocol`, `PhotoStorage` |
| 函数/变量 | lowerCamelCase | `analyzeImage()`, `currentCategory` |
| 常量 | lowerCamelCase | `maxRetryCount` |
| 枚举值 | lowerCamelCase | `case literature`, `case poetry` |
| 协议 | UpperCamelCase + 描述性后缀 | `AIServiceProtocol`, `StorageManaging` |

### 5.3 SwiftUI 规范

- View body 不超过 30 行，超过则拆分为子 View
- 使用 `@State`, `@Binding`, `@Environment` 时必须注释说明用途
- 样式统一从 `Shared/Theme/` 引用，不在 View 中硬编码颜色和字号

### 5.4 SwiftData 规范

- 所有 Model 定义在 `Core/Database/` 中
- Model 属性必须有注释说明用途
- 关系型数据必须定义反向关系

### 5.5 错误处理

- 网络请求必须处理超时、无网络、API 错误三种情况
- 用户可见的错误信息使用中文，不暴露技术细节
- 所有 AI 调用失败 → 文件留在 `未整理/`，toast 告知用户

### 5.6 注释规范

- 公开接口必须有文档注释（`///`）
- 复杂业务逻辑必须有行内注释说明"为什么"
- 不写无意义注释（如 `// 设置标题` 后面跟 `title = "标题"`）

## 六、测试规范

### 6.1 测试范围

| 层级 | 必须测试 | 可选 |
|------|---------|------|
| Core/Database | 数据模型 CRUD、关系、迁移 | - |
| Core/AI | Prompt 拼接、格式校验、归一化匹配 | API 集成测试 |
| Core/Storage | 文件读写、空间计算、图片缩略图 | - |
| Core/Keychain | Key 存取、删除 | - |
| Features/* | 核心交互逻辑 | UI 快照测试 |

### 6.2 测试命名

```swift
func test_[功能]_[场景]_[预期]() {
    // 示例
    func test_analyzeImage_networkTimeout_fileStaysInUnsorted()
    func test_renameFile_matchesXYFormat_promotesToRoot()
}
```

### 6.3 测试原则

- 每个模块至少覆盖：正常流程 + 边界情况 + 异常情况
- 不依赖网络的测试使用 Mock
- AI 相关测试使用 Mock AI Service，不实际调用 API

## 八、Git 规范

### 8.1 Commit Message 格式

```
<type>(<scope>): <description>

<body>

Co-Authored-By: Claude <noreply@anthropic.com>
```

### 8.2 Type 类型

| type | 含义 |
|------|------|
| feat | 新功能 |
| fix | 修复 bug |
| refactor | 重构（不改变功能） |
| style | 代码格式调整 |
| test | 添加或修改测试 |
| docs | 文档更新 |
| chore | 构建配置等杂项 |

### 8.3 Scope 范围

使用模块文件夹名：`database`, `storage`, `ai`, `collection`, `detail`, `settings`, `share-extension` 等

### 8.4 示例

```
feat(share-extension): 实现分享面板内的分类选择 UI

在 Share Extension 中添加四个分类按钮（找书/找诗/找画/找歌），
```

| 分支 | 用途 |
|------|------|
| `main` | 稳定版本，始终可编译运行 |
| `dev` | 开发主线 |
| `feature/模块名` | 单个模块的开发分支 |

### 8.6 Git 禁止事项

- 禁止提交 `.DS_Store`、`xcuserdata/` 等临时文件
- 禁止提交 API Key 或敏感信息
- 禁止向 `main` 直接推送，必须通过 `dev` 合并

- 选择或更换技术栈时

## 十、重构规范

### 10.1 重构前提

- 现有功能**测试全部通过**才能开始重构
- 重构不改变外部行为（功能不变）
- 重构范围必须明确，不扩大化

### 10.2 重构流程

1. 创建 ADR 记录重构原因和目标
2. 确保所有相关测试通过
3. 小步修改，每步都能编译运行
4. 每步修改后运行测试
5. 更新 ARCHITECTURE.md
6. 记录到 DEVELOPMENT_LOG.md
7. 提交时 type 使用 `refactor`

### 10.3 禁止事项

- 禁止在重构中顺便加新功能
- 禁止大范围一次性重构
- 禁止没有测试覆盖的重构

## 十一、Capture:D 专项规范

### 11.1 AI 安全规范

| 规则 | 说明 |
|------|------|
| AI 默认静默 | 打开 app 时 AI 不做任何工作 |
| 手动触发 | 仅在用户进入未整理文件夹并点击"AI 分析"按钮后启动 |
| 可选分析 | 用户选中具体图片才分析，非全选 |
| 费用上限 | 用户可设月度分析上限，到达后 app 停止发送 API 请求 |
| 跨分类省 token | 同一张图在多个大类有未整理副本时，一次调用一并分析 |
| 纯文本返回 | AI 返回内容使用纯文本格式，不用 Markdown，降低 token 消耗 |
| 格式校验 | AI 返回的出处名必须遵循 `作品名\|作者` 格式，不合规留在未整理 |

### 11.2 数据流向规范

```
入口 1：用户截屏/保存图 → Share Extension
       ↓
Share Extension 写入 App Group（图片 + 分类 + 可选命名 + 可选标签）
       ↓
主 App 启动时读取 → 命名格式正确则 sorted，否则 unsorted

入口 2：用户点击 App 内 FAB 按钮
       ↓
纯文字 → 碎碎念（直接 sorted）
带图片 → 选择分类（必选1+）+ 可选命名 + 可选标签 → 入库

两个入口之后：
       ↓
用户手动进入未整理文件夹 → 选图 → 点 AI 分析
       ↓
AI 返回出处名 + 内容 → 校验格式 → 移出未整理 → 正常命名归档
       ↓
瀑布流中可见
```

### 11.3 文件命名规范

| 场景 | 格式 | 示例 |
|------|------|------|
| 正常归档 | `作品名\|作者` | `斗破苍穹\|天蚕土豆` |
| 找画（画师） | `画师名\|平台名` | `某画师\|Pixiv` |
| 找画（未知） | `未知风格-编号` | `未知风格-001` |
| 书单 | `日期+书单` | `2026-07-15 书单` |
| 歌单 | `日期+歌单` | `2026-07-15 歌单` |
| 未整理 | 存入时间 | `2026-07-15 14:30` |

**全局禁止：** 书名号 `《》`、除 `|` 以外的特殊分隔符

### 11.4 未整理文件提升规范

- 用户手动改文件名为 `X|Y` 格式 → 保存瞬间系统自动提升到大类根目录
- 目标位置已有同名文件 → 自动合并（按时间线粘贴内容，用户编辑内容保留）
- 提升检测为本地正则匹配逻辑，不需要 AI
- AI 只被允许访问 `未整理/` 文件夹内的内容

### 11.5 Share Extension 规范

- 只接收图片，非图片内容自动过滤
- 在分享面板内完成分类选择（必选1+）、可选命名和标签输入，**不跳转到主 app**
- 受 iOS 限制（内存约 120MB、运行时间几秒），只做存图 + 记录分类/命名/标签
- 不在 Share Extension 中调用 AI
- 标签为手动输入，无历史补全（因 Extension 无法访问 SwiftData）

### 11.6 UI/UX 规范

| 项目 | 规范 |
|------|------|
| 整体风格 | 轻盈、干净、无 AI 感，接近精致的个人收藏夹 |
| 时间标记 | Threads 风格左对齐相对时间（X分钟前/X小时前），无气泡底色 |
| 动画 | Safari 浏览器物理弹性效果 |
| 筛选动画 | 选中卡片弹动变大 + 未选中碎片化消失 |
| 浮层卡片 | 图片撑满无白边，日期为左下角悬浮白字 |
| AI 注释 | 淡灰小字，像手写备注，不出现"AI 分析结果" |
| 文本编辑 | 逐条 ContentBlock 铅笔按钮编辑 + 底部追加按钮 |
| 弹窗高度 | 统一 45% 屏高 |
| 搜索 | 内嵌首页，搜索+分类可叠加过滤 |
| 显示模式 | 仅浅色模式 |
| 设备支持 | 仅 iPhone |

### 11.7 回收站规范

- 入口在设置页
- 30 天后自动永久删除（学习 iPhone 照片 app）
- 恢复时直接恢复原样（含文本内容、图片引用、分类归属）
- 原文件已不存在时显示"文件已不存在"
- 跨分类图片删除只删当前分类引用，图片文件保留
